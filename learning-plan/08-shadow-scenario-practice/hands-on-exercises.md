# Hands-On Exercises: Shadow & Scenario Practice

---

## Exercise 1: Full Scenario Run — Timed

**Objective:** Practice diagnosing and resolving all 15 common issues under time pressure.

1. Set up a fresh VM or reset your current one
2. For each of the 15 scenarios in the reference material:
   - Set up the scenario (break the system)
   - Start a timer
   - Diagnose and resolve the issue
   - Stop the timer
   - Document what you did
   - Score yourself using the rubric
3. After completing all 15:
   - Calculate your average score
   - Identify your weakest areas
   - Re-practice the weakest 3 scenarios
4. Target: Complete all 15 scenarios in under 3 hours total (average 12 min each)

## Exercise 2: Write First-Contact Scripts

**Objective:** Develop your own communication templates.

For each of the top 5 issues, write your own first-contact script:
1. What questions will you ask first?
2. In what order?
3. How will you explain the fix to the user?
4. What follow-up will you provide?

Write these in your own words — don't copy the templates from the reading material. Make them sound like you.

Have a colleague or mentor review them for:
- Clarity
- Appropriateness for non-technical users
- Completeness
- Professionalism

## Exercise 3: Shadow a Support Session

**Objective:** Observe real support work.

If possible:
1. Arrange to shadow an experienced supporter for at least 2 hours
2. Use the shadowing checklist from the reference material
3. Take notes on every ticket observed
4. After the session, debrief with the supporter using the debrief questions
5. Document your key takeaways

If shadowing isn't available:
1. Browse r/techsupport or Spiceworks community
2. Read through real support tickets/threads
3. For each thread:
   - What would your first question be?
   - What diagnostic steps would you take?
   - Was the solution correct?
   - What would you have done differently?

## Exercise 4: Build Your Ticket Template

**Objective:** Create a standardized ticket documentation format.

Create a template that includes:
1. **Issue Description**: User's words, verbatim if possible
2. **Impact**: One user? Multiple users? Department-wide?
3. **Urgency**: Low, Medium, High, Critical (define what each means)
4. **Investigation**: Steps taken, commands run, findings
5. **Root Cause**: What was actually wrong
6. **Resolution**: What fixed it
7. **Verification**: How you confirmed it's fixed
8. **Follow-up**: Any pending actions or monitoring needed
9. **Category**: Which of the 15 issue types
10. **Time Spent**: Total time from open to close

Fill in this template for at least 3 practice scenarios.

## Exercise 5: Technical-to-Plain-Language Translation Practice

**Objective:** Practice explaining technical issues in user-friendly terms.

For each technical finding below, write a 1-2 sentence explanation a non-technical user would understand:

1. "Your DNS cache is corrupted and needs to be flushed."
2. "Your Windows user profile is corrupted and loading a temporary profile."
3. "The Print Spooler service has crashed and needs to be restarted."
4. "Your Outlook OST file is corrupted and needs to be rebuilt."
5. "Your device isn't compliant with Intune policies, which is blocking access."
6. "A recent Windows Update has a known bug causing BSOD with your display driver."
7. "OneDrive Known Folder Move has redirected your Desktop to the cloud."
8. "Your Kerberos tickets have expired and you need to re-authenticate."

Compare your explanations with the reference material's translation guide. Refine.
